# End-to-end deployment runbook (Flux GitOps model)

Manual per-layer flow. Each `terraform apply` is independent — you can
stop between any two layers, inspect, and resume.

## What you need before starting

### Tools (install once)

- `terraform` (>= 1.7)
- `kubectl`
- `kubeseal` ([install](https://github.com/bitnami-labs/sealed-secrets#kubeseal))
- `openssl`, `jq`, `yq`, `curl` (`tf.sh` will install missing ones via brew)
- `helm` (TF helm provider; tf.sh installs)
- `aws` CLI (for S3 backend + SM)

### Infrastructure prerequisites

- A Harvester cluster you can reach + its kubeconfig file on disk
- A Rancher hostname you control (DNS record will be pointed at the
  LB IP layer 01 allocates)
- A dc-api hostname + cloud-ui hostname (DNS records will be pointed
  at the LB IP layer 04 allocates)
- AWS account with:
  - An S3 bucket for Terraform state (the layers use the bucket
    referenced in each layer's `versions.tf`)
  - A DynamoDB table for state locks (same)
  - Secrets Manager entries for the `*.secret.tfvars.template`
    placeholders each layer carries — `tf.sh` reads these at apply
    time. The minimum set:
      `<org>-dc/<env>/github_token` (any path you choose;
      match what your `secrets.secret.tfvars.template` references)

### Credentials you'll create + store

| Credential | Where created | What it's used for | Where it lives |
|---|---|---|---|
| **AWS access key + secret** | AWS IAM console | S3 backend + DynamoDB lock + Secrets Manager fetch | `~/.aws/credentials` or env vars on operator workstation |
| **Asgardeo M2M client_id + secret** | Asgardeo console → Applications → Create M2M | `provider "asgardeo"` in layer 03 — creates the OIDC apps for Rancher SSO + cloud-ui SPA + cloud-ui BFF | `03-asgardeo-auth/secrets.secret.tfvars` (via SM template) |
| **GitHub personal-access token** | https://github.com/settings/tokens | Once: layer 06 registers Flux's SSH deploy key on the consumer repo via GitHub API. Scope: `repo` (or fine-grained equivalent with Administration: read/write on the target repo). | `TF_VAR_github_token` env var OR Secrets Manager path referenced by `06-flux-bootstrap/secrets.secret.tfvars.template`. **Not persisted on the cluster.** Can be rotated/revoked after the bootstrap completes — only the resulting SSH key is needed long-term. |
| **GHCR personal-access token** | https://github.com/settings/tokens — `read:packages` scope (`write:packages` if your CI also pushes to GHCR) | Lets the in-cluster pods pull container images from ghcr.io. Skip if you only ever pull public images (e.g. `ghcr.io/wso2/dc-api`). | Sealed into the cluster by the wizard as `ghcr-pull-secret`. |
| **Harvester admin kubeconfig** | Downloaded from Harvester UI → Support page | Layer 00 → Rancher VM creation. Layer 02 → Harvester→Rancher integration. Layer 06 → SealedSecret content for dc-api's `DCAPI_HARVESTER_KUBECONFIG`. | A file path on your workstation; passed to layers via the harvester provider and to the wizard via prompt. |
| **Rancher admin token** | Auto-generated by layer 01 (`rancher2_bootstrap`); not something you create yourself | Every downstream layer that talks to Rancher (02, 04, 05, 06). Wizard reads it for the dc-api `DCAPI_RANCHER_TOKEN` secret. | Layer 01's TF state output. Fetched via `./tf.sh output --region <env> --layer rancher-auth`. |
| **TLS cert + key for `dc-api` / `cloud-ui` ingresses** | Optional. Either bring your own (signed by a CA your users trust) or let the wizard generate a self-signed pair (1-yr SAN cert covering both hostnames) | Sealed into the cluster as the `dc-api-tls` Secret both Ingresses reference. | Generated at wizard time OR PEM files you point the wizard at. |

### What gets auto-created (you don't have to do anything for these)

- **SSH deploy key for Flux** — generated by layer 06 via `tls_private_key` resource, registered on the consumer repo via the GitHub provider, installed in the cluster as a Secret in `flux-system` namespace.
- **Postgres password** — generated by the wizard at `--seal-secrets` time (24-char hex), sealed into `dc-postgres-secret`. Stable across re-runs.
- **BFF session secret** — generated by the wizard at `--seal-secrets` time (32-byte base64 AES key), sealed into `dc-api-secrets` under `DCAPI_BFF_SESSION_SECRET`. Stable across re-runs; rotating invalidates every active BFF session.
- **Self-signed CA chain for `dc-webhook`** — generated inside TF state by the existing `modules/management/dc-webhook` module via `hashicorp/tls`. 10-year validity, regenerated only on TF state rebuild.

## Repos involved

- `wso2/open-cloud-datacenter` — the library (TF modules + `flux/`
  manifests + `scripts/init-flux.sh` wizard). Reference it as
  `modules/open-cloud-datacenter/` inside your consumer repo via
  `tf.sh` + `dependencies.yaml`.
- Your **consumer repo** — holds your `environments/<env>/` TF layer
  configs + the Flux overlay generated by the wizard.

## The flow at a glance

```
┌─ TERRAFORM (Harvester host cluster) ────────────────┐
│  layer 00  bootstrap        Rancher VM on Harvester │
│  layer 01  rancher-auth     admin token             │
│  layer 02  management       Harvester → Rancher     │
│              + namespace-credential-provisioner     │
│              + KubeOVN install + external network   │ ← NEW (modules/management/kubeovn)
│  layer 03  asgardeo-auth    OIDC apps on IdP        │
│  layer 04  dc-controlplane  3-node RKE2 cluster     │
│  layer 05  dc-operators     dc-webhook on Harvester │ ← stays TF (Harvester-side)
│  layer 06  flux-bootstrap   Flux on dcapi-cluster   │
│                             pointed at this repo    │
└─────────────────────┬───────────────────────────────┘
                      │
                      │  Flux Source Controller (on dcapi-controlplane-rke2)
                      │  polls every 1m, picks up commits to
                      │  environments/<env>/flux/
                      ▼
┌─ FLUX (dcapi-controlplane-rke2 cluster) ────────────┐
│  STAGE A: infrastructure                            │
│   - sealed-secrets controller                       │
│   - cert-manager                                    │
│   - ingress-nginx                                   │
│                                                     │
│  >>> WIZARD: re-run with --seal-secrets             │
│      writes sealed-*.yaml, commits, push            │
│                                                     │
│  STAGE B: platform                                  │
│   - dc-postgres                                     │
│   - dc-api                                          │
│   - cloud-ui                                        │
│   - keyvault-operator                               │
└─────────────────────────────────────────────────────┘
```

## Two cluster targets, two ownership models

Cluster | Owner | What lives here
---|---|---
**Harvester host** (`harvester-dev` kubectl context) | Terraform | KubeOVN, namespace-credential-provisioner, dc-webhook (intercepts KubeVirt VM admission — only meaningful on Harvester), Harvester networks/storage
**dcapi-controlplane-rke2** (RKE2 cluster running INSIDE Harvester as VMs) | Flux | dc-api, cloud-ui, dc-postgres, keyvault-operator, the in-cluster add-ons (sealed-secrets, cert-manager, ingress-nginx)

dc-webhook does NOT live in Flux. Its `MutatingWebhookConfiguration`
intercepts `kubevirt.io/VirtualMachine` admission, and KubeVirt only
runs on the Harvester host cluster. The TF module
`modules/management/dc-webhook` already hand-rolls its TLS via the
`hashicorp/tls` provider (no cert-manager dependency on the Harvester
side).

## ARC retirement

Today, ARC (Actions Runner Controller) runs ephemeral GitHub runners
inside the dcapi-controlplane cluster, and CI does `kubectl set image`
through them. Under the Flux model:

- CI builds + pushes container images to GHCR (any runner — including
  GitHub-hosted — works; no in-cluster runners needed).
- Flux Image Automation Controller watches GHCR and updates the
  Deployment manifest. Pod rolls.
- The `kubectl set image` path is dead.

ARC stays in the cluster only if the consumer keeps using it for
out-of-band tasks (e.g. running other CI jobs that need privileged
in-cluster access). If you keep it, the existing `dc-api-deployer`
ServiceAccount + Role + RoleBinding in `dc-system` won't auto-survive
Flux's `prune: true` — re-add them to your platform-overlay if needed.

## Step-by-step

### 1. Pull OCD into your consumer repo

The consumer repo's `dependencies.yaml` should pin OCD:

```yaml
versions:
  - src: https://github.com/wso2/open-cloud-datacenter.git
    tag: spike/flux-gitops      # bump to a release tag once published
```

Then `./tf.sh init` (or however your wrapper resolves modules) populates
`environments/<env>/modules/open-cloud-datacenter/`.

### 2. Tear down the old in-cluster TF (if migrating)

If you're moving an existing env from the old TF-managed in-cluster
model to Flux, destroy the layers that managed in-cluster workloads
**before** running layer 06. The cluster itself stays.

```bash
cd environments/<env>/02-dc-controlplane-services
terraform destroy

cd ../02-dc-operators
terraform destroy
```

Optionally also destroy + reapply `02-dc-controlplane` for a truly
clean cluster — quicker than `kubectl delete -A everything` for
verifying the new flow.

### 3. (Re-)apply the base infra layers

Standard sequence:

```bash
cd environments/<env>/00-bootstrap                && terraform apply
cd ../01-rancher-auth                             && terraform apply
cd ../02-management                               && terraform apply   # ← see KubeOVN note below
cd ../03-asgardeo-auth                            && terraform apply
cd ../02-dc-controlplane                          && terraform apply
cd ../02-dc-operators                             && terraform apply   # ← dc-webhook on Harvester (kept as TF)
```

**KubeOVN note for layer 02-management:** the new
`modules/management/kubeovn` module should be wired into this layer
alongside the existing `harvester-integration` + `namespace-credential-provisioner`
calls. If KubeOVN is already installed on the cluster (brownfield case
— hand-installed manually), set `manage_kubeovn_install = false` in the
module call so TF doesn't try to install a second copy. See the
module's README for the brownfield-import recipe to adopt the existing
Helm release + ProviderNetwork + Vlan + Subnet resources.

At the end you have:
- Rancher running
- Harvester registered into Rancher
- KubeOVN installed + external ProviderNetwork/Vlan/Subnet ready
- OIDC apps registered on Asgardeo (rancher-sso, cloud-ui SPA,
  cloud-ui-bff confidential)
- `dcapi-controlplane-rke2` cluster Ready (3 nodes if HA)
- dc-webhook running on the Harvester host cluster (intercepts
  KubeVirt VirtualMachine admission)

### 4. Render the Flux overlay (without secrets)

```bash
# From the consumer repo root:
./environments/<env>/modules/open-cloud-datacenter/scripts/init-flux.sh \
    --consumer-dir . \
    --env-name <env>
```

The wizard prompts for hostnames + Asgardeo org + GHCR org + VPC
values, writes `environments/<env>/flux/` with placeholders
substituted. Commit + push:

```bash
git add environments/<env>/flux
git commit -m "Add Flux overlay for <env>"
git push
```

### 5. Apply layer 06 (Flux bootstrap)

```bash
cd environments/<env>/06-flux-bootstrap

# Provide your GitHub PAT once — used to register Flux's deploy key
# on the consumer repo. Not stored on the cluster.
export TF_VAR_github_token="ghp_xxxxxxxxxxxxx"

terraform init
terraform apply
```

This:
- Installs Flux controllers into the `flux-system` namespace.
- Generates an SSH deploy key, registers it on the consumer fork.
- Commits Flux's `gotk-components.yaml` + `gotk-sync.yaml` into
  `environments/<env>/flux/flux-system/` on the consumer repo.
- Flux's root Kustomization then auto-reconciles
  `environments/<env>/flux/` from the consumer repo.

### 6. Watch Stage A come up

```bash
export KUBECONFIG=<path-to-dcapi-controlplane-kubeconfig>
kubectl --namespace=flux-system get kustomizations -w
```

Expect (in order):
- `flux-system` Ready (root sync)
- `infrastructure` Ready (cert-manager, sealed-secrets, ingress-nginx)
- `platform` will be **NotReady** with a "sealed secrets not found"
  reconciliation error — expected; secrets aren't sealed yet.

### 7. Seal secrets

You need these values handy:

| Value | Where to get it |
|---|---|
| Rancher admin token | `terraform output -raw admin_token` in layer 01 |
| Harvester cloud_credential_id | `terraform output cloud_credential_id` in layer 02 |
| Harvester kubeconfig path | the file you've been pointing layer 00's `harvester_kubeconfig` at |
| Asgardeo BFF client_id + secret | `terraform output -raw bff_client_id` / `bff_client_secret` in layer 03 |
| Asgardeo rancher-sso client_id | `terraform output -raw client_id` in layer 03 |
| Asgardeo cloud-ui SPA client_id | `terraform output -raw cloud_ui_client_id` in layer 03 |
| GHCR PAT | https://github.com/settings/tokens — `read:packages` scope |

Then re-run the wizard with `--seal-secrets`:

```bash
./environments/<env>/modules/open-cloud-datacenter/scripts/init-flux.sh \
    --consumer-dir . \
    --env-name <env> \
    --seal-secrets \
    --force
```

The wizard:
- Fetches the sealed-secrets controller's public cert from the cluster.
- Generates random Postgres password + BFF session secret locally
  (re-runs do NOT rotate them — manual edit + re-seal if rotation is
  wanted).
- Prompts for the TLS source for `dc-api-tls`: `[s]elf-signed`
  (wizard runs openssl, 1yr cert with both Ingress hostnames in SANs)
  or `[b]yo` (paths to existing PEM files).
- Seals 4 secrets:
  - `dc-postgres-secret` — postgres password
  - `dc-api-secrets` — composite (DCAPI_DB_URL, OIDC_AUDIENCE,
    RANCHER_TOKEN, RANCHER_HARVESTER_CREDENTIAL, OPERATOR_*,
    BFF_CLIENT_ID/SECRET/SESSION_SECRET, HARVESTER_KUBECONFIG)
  - `dc-api-tls` — `kubernetes.io/tls` Secret both Ingresses reference
  - `ghcr-pull-secret` — dockerconfigjson for GHCR pulls
- Drops them as `sealed-*.yaml` next to the overlay.
- Rewrites `platform-overlay/kustomization.yaml` to reference them.

Commit + push:

```bash
git add environments/<env>/flux
git commit -m "Seal <env> secrets"
git push
```

### 8. Watch Stage B come up

```bash
# Against dcapi-controlplane-rke2 (KUBECONFIG already set from step 6)
kubectl --namespace=flux-system get kustomizations -w
# wait for "platform" to flip to Ready

kubectl --namespace=dc-system get pods -w
# dc-postgres-0     Running
# dc-api-xxxxx      Running
# cloud-ui-xxxxx    Running

kubectl --namespace=keyvault-system get pods
# keyvault-controller-manager-xxxxx  Running

# dc-webhook lives on the HARVESTER host cluster (different kubeconfig):
KUBECONFIG=<harvester-kubeconfig> kubectl --namespace=dc-webhook get pods
# dc-api-webhook-xxxxx  Running
```

### 9. Verify

```bash
curl -k https://<dcapi-hostname>/healthz
# 200 OK

open https://<cloud-ui-hostname>
# OIDC login flow → cloud-ui shell
```

## Iterating after first install

- **Code change to dc-api**: push to dc-api source repo → GHA workflow
  builds + pushes `ghcr.io/<your-org>/dc-api:main-<sha>` → Flux Image
  Automation Controller bumps the manifest → rolling update. ~3 min.
- **In-cluster config change**: edit YAML under
  `environments/<env>/flux/` → commit + push → Flux reconciles. ~30s.
- **Infra change** (cluster nodes, network, …): edit the relevant TF
  layer → `terraform apply`. Same as before — TF still owns base infra.

## Troubleshooting

```bash
# What's Flux doing right now?
flux get all -A

# A Kustomization stuck?
flux logs --kind=Kustomization --name=platform

# A sealed secret failing to decrypt?
kubectl --namespace=dc-system get sealedsecrets
kubectl --namespace=sealed-secrets logs deploy/sealed-secrets-controller

# Image automation not bumping?
flux get image all -A
flux logs --kind=ImageUpdateAutomation --name=...
```
